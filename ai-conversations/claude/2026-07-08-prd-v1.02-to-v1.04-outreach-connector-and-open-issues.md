Summary: Marc brought Claude the LeadPilot PRD (v1.01) and asked what
it would take to eventually support spreadsheet formats beyond Google
Sheets. That opened into a full working session: Claude found a real
gap in v1.01 (lead-outreach tools were required by the user stories

replace the Google-Voice-dependent history tool, and closing out the
rep-approval-token mechanism (Issue 003) with a concrete, crypto-free
design. Governance (equity split) and a compliance gap (rep-approval
≠ legal compliance) were resolved along the way.

Key decisions (see decisions/README.md #014–021 for the canonical log):
- Lead outreach (call/text/email) follows the same rep-approval gate
  as everything else — no autonomous-send carve-out, even though it
  was tempting to treat outreach differently from spreadsheet writes.
- `initiate_lead_call` never touches Google Voice or any calling app.
  Confirmed directly (web search) that Google Voice has no official
  API and its Acceptable Use Policy explicitly prohibits scripting or
  automating call/message placement — even a script that stops short
  of dialing. Rep approval copies the number to the clipboard instead;
  the rep dials manually.
- `initiate_backoffice_call` is retired — back-office handoffs are
  always a Slack message now, including for what used to be the "call"
  option, and stay rep-approved with no urgency exception.
- `get_contact_history` reads from a LeadPilot-owned, append-only
  contact-history log instead of any external API — this also gave
  Issue 003 (approval mechanism) a home: the same log row's `stage`
  field, gated by an atomic conditional update, is both the audit
  trail and the enforcement mechanism. No signed or opaque token needed.
- `log_call_outcome` exists because calls now happen entirely outside
  LeadPilot — without a rep reporting the outcome back, Rank 3's
  "unanswered call → follow up" rule has no data for calls specifically.
- Google Docs is deferred as a lead-source connector — it doesn't
  share the tabular row/column shape Sheets/Excel/OnlyOffice have in
  common, and needs its own data-modeling decision first.
- Equal 50/50 ownership between Marc and Abdoul; either partner may
  fork or use the work as they see fit (still flagged as needing a
  real written agreement eventually).
- Rep-approval gating reduces bulk/autodialer-style liability but does
  not by itself satisfy TCPA, CAN-SPAM, do-not-call scrubbing, or the
  `search_communications` data-exposure review — those remain open
  compliance items regardless of who clicks approve.
- **Tech stack locked (Decision 022, supersedes Decision 005):**
  Python + Claude Agent SDK for the tool-calling loop; FastAPI +
  Jinja2/htmx for the dashboard (no separate SPA); Postgres via Neon
  for the state store — not SQLite, corrected mid-conversation once it
  became clear the always-on dashboard and the hourly batch job are
  separate containers that both need to read/write the same state, so
  a local SQLite file wouldn't be visible to both; Render Cron Job
  (hourly) for the batch run plus a separate Render Web Service for
  the dashboard, sharing the same Postgres instance; Google API
  Python client (Sheets/Drive/Gmail), Twilio (SMS), `slack_sdk`
  (Slack); GitHub Actions running the eval suite per PR.
